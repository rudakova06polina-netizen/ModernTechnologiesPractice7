# Часть 5: Spring Security

## Задание 5.1

1. Войдите как `user / password`. Можно ли выполнить `POST /api/students`? Какой код?

Ответ: Нет, нельзя. Пользователь с ролью USER имеет только право читать данные, но не создавать новые. При попытке отправить POST запрос сервер вернёт HTTP-код 403 Forbidden. Это значит, что пользователь аутентифицирован, но у него недостаточно прав для выполнения этого действия. В типовой конфигурации SecurityConfig для USER разрешены только GET запросы, а POST, PUT и DELETE требуют роль ADMIN.

2. Войдите как `admin / password`. Получается ли выполнить `POST /api/students`?

Ответ:  Да, получается. Администратор имеет роль ADMIN, которой в конфигурации безопасности разрешены все методы: и GET, и POST, и PUT, и DELETE.

3. Что происходит при `GET /api/students` без аутентификации?

Ответ: Запрос не проходит, сервер возвращает HTTP-код 401 Unauthorized. Это значит, что пользователь не аутентифицирован, то есть система не знает, кто он. Spring Security перехватывает запрос, видит, что ресурс защищён, и не пускает анонимного пользователя.

## Задание 5.2

1. Различие между правилами в `SecurityConfig` и `@PreAuthorize`?

Ответ: SecurityConfig работает на уровне HTTP-запросов. @PreAuthorize работает на уровне методов.

2. Что будет при конфликтующих правилах?

Ответ: Spring Security применяет наиболее строгое правило. Если в SecurityConfig вы разрешили USER доступ к /api/students, но на методе контроллера поставили @PreAuthorize("hasRole('ADMIN')"), то пользователь USER не пройдёт — методная аннотация всё равно заблокирует вызов. @PreAuthorize проверяется позже и может ужесточить правила. А вот смягчить правила на уровне метода не получится: если фильтр сказал "запрещено", метод даже не вызовется. Правила не конфликтуют в прямом смысле — они работают как цепочка. Каждое звено может отказать в доступе, и этого достаточно для запрета.
